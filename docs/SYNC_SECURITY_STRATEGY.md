# Стратегия безопасности синхронизации документов

## Обзор

Документ описывает стратегию обеспечения безопасности системы синхронизации текстовых документов в Noetec. Рассматриваются криптографические механизмы защиты от подделки записей, несанкционированного доступа и атак на целостность данных.

## Текущая архитектура синхронизации

### Структура хранения

```
.sync/pages/<encoded-path>/<deviceId>.oplog.jsonl   — общий лог (WebDAV/Dropbox)
.noetec/device.json                                  — локальный идентификатор устройства
```

### Формат записи OpLogEntry

Каждая запись содержит:
- `hlc` — гибридная логическая метка времени (physicalMs, counter, deviceId)
- `parent` — ссылка на предыдущую запись (линейная история)
- `mergeParents` — массив дополнительных родителей для merge-операций
- `blockOps` — операции над блоками (insert, delete, update, move)
- `deviceId` — идентификатор устройства-автора
- `fileHash` — хеш состояния файла (опционально)

### Принцип работы

1. Каждое устройство пишет **только в свой файл** `<deviceId>.oplog.jsonl`
2. Формат JSONL — append-only, построчный
3. DAG строится из всех device-файлов через `OpLogDag.fromEntries()`
4. Конфликты разрешаются через merge-операции с массивом `mergeParents`, содержащих родителей всех сливаемых веток

### Ключевое преимущество

**Бесконфликтная файловая синхронизация**: поскольку каждое устройство пишет в отдельный файл, на уровне файловой системы (WebDAV/Dropbox) конфликтов не возникает. Конфликты переносятся на уровень логики (CRDT-merge).

## Анализ угроз

### Угроза 1: Подделка записей от чужого имени

**Сценарий**: Злоумышленник с доступом к общей папке может открыть чужой файл `<victimDeviceId>.oplog.jsonl` и дописать в него свои записи, указав `deviceId` жертвы.

**Последствия**: Документ будет содержать изменения, якобы сделанные другим устройством. Это нарушает атрибуцию изменений и может привести к нежелательным merge-результатам.

**Текущая защита**: Отсутствует. `OpLogDag.fromEntries()` принимает все записи без верификации подлинности.

### Угроза 2: Несанкционированное участие в документе

**Сценарий**: Любой, кто получил доступ к общей папке (например, через шаринг Dropbox), может создать файл `<randomId>.oplog.jsonl` и начать «редактировать» документ от нового устройства.

**Последствия**: Появление нежелательных участников с возможностью влиять на содержимое документа.

**Текущая защита**: Отсутствует. Нет механизма авторизации устройств.

### Угроза 3: Усечение истории (откат)

**Сценарий**: Злоумышленник может удалить последние N строк из чужого `.oplog.jsonl` файла и дописать свои записи с поддельным `deviceId`.

**Последствия**: Потеря легитимной истории изменений. Если перекрёстные ссылки отсутствуют, усечение может остаться незамеченным.

**Текущая защита**: Частичная. Перекрёстные ссылки через `mergeParents` возникают только при merge-операциях, а не при каждом edit.

### Угроза 4: Replay attack

**Сценарий**: Злоумышленник копирует валидную подпись из одного документа и вставляет в другой (если подпись не привязана к documentId).

**Последствия**: Подделка записей с использованием легитимных подписей из других контекстов.

**Текущая защита**: Отсутствует.

### Угроза 5: HLC-спуфинг

**Сценарий**: Злоумышленник выставляет `physicalMs` далеко в будущее, чтобы его записи всегда «побеждали» при merge (HLC compare — `physicalMs` первый в сортировке).

**Последствия**: Манипуляция порядком применения операций, потенциально — захват контроля над merge-результатами.

**Текущая защита**: Отсутствует.

### Угроза 6: Подмена файла целиком

**Сценарий**: Злоумышленник заменяет ваш `<yourDeviceId>.oplog.jsonl` на полностью свой файл с валидными подписями от своего ключа, но с вашим deviceId в имени.

**Последствия**: Полная подмена истории изменений от вашего имени.

**Текущая защита**: Отсутствует.

## Рекомендуемая стратегия защиты

### Фаза 1: Базовая криптографическая подпись

**Цель**: Закрыть угрозы 1 (подделка записей) и частично 3 (усечение).

#### Механизм

1. **Генерация ключевой пары** при первом запуске устройства:
   - Алгоритм: Ed25519
   - Приватный ключ: хранится в `flutter_secure_storage` (не в файловой системе)
   - Публичный ключ: сохраняется в `device.json` и в первой записи oplog-файла

2. **Самосертифицируемый deviceId** (опционально):
   - `deviceFingerprint = SHA256(pubKey)`
   - Используется для верификации, не для HLC/именования файлов
   - **Важно**: сохранить существующий `deviceUuid` для HLC и файлов, чтобы не ломать совместимость

3. **Подпись каждой записи**:
   - Подписывается: `canonicalJson(entry_without_signature) + documentPath`
   - Поля подписи: `signature: String`, `pubKey: String?` (только в первой записи)
   - Подпись добавляется в `OpLogEntry` и сериализуется через `OpLogSerializer`

4. **Верификация при чтении**:
   - `OpLogDag.fromEntries()` проверяет подпись каждой записи
   - Первая запись файла должна содержать `pubKey`
   - Если подпись невалидна — запись отклоняется, вся цепочка после нарушения отвергается

#### Изменения в архитектуре

| Компонент | Изменение | Файл |
|---|---|---|
| `DeviceIdentity` | + `publicKey: String` | `entity/device/device_identity.dart` |
| Новый: `ICryptoService` | Ed25519 sign/verify, SHA-256 | `service/crypto_service.dart` |
| `OpLogEntry` | + `signature: String`, + `pubKey: String?` | `systems/oplog_system/oplog_models.dart` |
| `OpLogSerializer` | + encode/decode `sig`, `pub` полей | `systems/oplog_system/oplog_serializer.dart` |
| `OpLogDag` | Верификация подписей при `fromEntries()` | `systems/oplog_system/oplog_dag.dart` |
| `OpLogWriter` | Подпись перед записью | `systems/oplog_system/oplog_writer.dart` |

#### Зависимости

- `package:cryptography` — для Ed25519 и SHA-256
- `flutter_secure_storage` — для хранения приватного ключа

#### Что закрывает

- ✅ Угроза 1: Подделка записей (невозможно без приватного ключа)
- ⚠️ Угроза 3: Усечение (частично — детектируется разрыв цепочки, но не всегда)
- ❌ Угроза 2: Несанкционированное участие (не закрывает)
- ❌ Угроза 4: Replay attack (не закрывает)
- ❌ Угроза 5: HLC-спуфинг (не закрывает)
- ❌ Угроза 6: Подмена файла целиком (не закрывает)

### Фаза 2: Авторизация устройств

**Цель**: Закрыть угрозу 2 (несанкционированное участие).

#### Механизм

1. **Манифест авторизованных устройств**:
   - Файл: `.sync/.manifest.json`
   - Содержит список устройств с их `pubKey` и подписями владельца документа
   - Формат:
     ```json
     {
       "owner_device_uuid": "xxx",
       "authorized_devices": [
         {
           "uuid": "yyy",
           "pubKey": "...",
           "added_by": "owner_uuid",
           "signature": "..."
         }
       ],
       "manifest_signature": "..."
     }
     ```

2. **Процесс добавления устройства**:
   - Владелец документа подписывает публичный ключ нового устройства
   - Запись добавляется в манифест
   - Манифест подписывается владельцем

3. **Фильтрация при построении DAG**:
   - `OpLogDag.fromEntries()` отклоняет записи от устройств, отсутствующих в манифесте
   - Если манифест отсутствует — документ считается публичным (обратная совместимость)

#### Изменения в архитектуре

| Компонент | Изменение | Файл |
|---|---|---|
| Новый: `ManifestStore` | Загрузка/сохранение манифеста | `systems/sync_system/manifest_store.dart` |
| `OpLogDag` | Фильтрация по манифесту | `systems/oplog_system/oplog_dag.dart` |
| UI | Интерфейс управления устройствами | `view/...` |

### Фаза 3: Перекрёстные ссылки (Witnesses)

**Цель**: Закрыть угрозу 3 (усечение истории) полностью.

#### Механизм

1. **Поле `seen` в каждой записи**:
   - Формат: `Map<String, String>` — `{deviceId: lastSeenHlcKey}`
   - При создании записи устройство указывает последние известные HLC от всех других устройств
   - Это создаёт полноценный witness-граф, а не только при merge

2. **Проверка целостности**:
   - При чтении чужого файла проверяется, что все записи, на которые ссылаются `seen`, действительно присутствуют
   - Если файл был откачен — ссылки из `seen` других устройств «повиснут»
   - Обнаружение dangling reference = обнаружение атаки

3. **Миграция существующих записей**:
   - Обратная совместимость: если `seen` отсутствует — запись считается валидной
   - Новые записи всегда включают `seen`

#### Изменения в архитектуре

| Компонент | Изменение | Файл |
|---|---|---|
| `OpLogEntry` | + `seen: Map<String, String>?` | `systems/oplog_system/oplog_models.dart` |
| `OpLogSerializer` | + encode/decode `seen` | `systems/oplog_system/oplog_serializer.dart` |
| `OpLogWriter` | Заполнение `seen` при записи | `systems/oplog_system/oplog_writer.dart` |
| `OpLogDag` | Проверка witness-целостности | `systems/oplog_system/oplog_dag.dart` |

#### Что закрывает

- ✅ Угроза 3: Усечение истории (полностью)
- ❌ Остальные угрозы не закрываются

### Фаза 4: TOFU и защита от HLC-спуфинга

**Цель**: Закрыть угрозы 4, 5, 6.

#### Механизм 1: Trust-on-First-Use (TOFU)

1. **Локальный кеш доверенных ключей**:
   - Файл: `.noetec/trusted_keys.json`
   - При первом наблюдении устройства его `pubKey` сохраняется локально
   - При последующих sync проверяется, что `pubKey` в первой записи совпадает с сохранённым
   - Если не совпадает — предупреждение пользователя

2. **Защита от подмены файла целиком**:
   - Злоумышленник не может заменить файл, так как `pubKey` в первой записи не совпадёт с сохранённым в `trusted_keys.json`

#### Механизм 2: HLC drift validation

1. **Проверка физического времени**:
   - При чтении чужих записей проверяется: `entry.hlc.physicalMs <= localTime + maxDrift`
   - `maxDrift` = 60 секунд (настраиваемо)
   - Если опережает — запись помечается как подозрительная или отклоняется

2. **Логирование аномалий**:
   - Записи с подозрительным HLC логируются для ручного анализа

#### Изменения в архитектуре

| Компонент | Изменение | Файл |
|---|---|---|
| Новый: `TrustedKeysStore` | TOFU-кеш | `service/trusted_keys_store.dart` |
| `SyncWatcher` | TOFU-проверка при первом наблюдении | `systems/sync_system/sync_watcher.dart` |
| `OpLogDag` | HLC drift validation | `systems/oplog_system/oplog_dag.dart` |

#### Что закрывает

- ✅ Угроза 4: Replay attack (частично — если подпись включает documentPath)
- ✅ Угроза 5: HLC-спуфинг
- ✅ Угроза 6: Подмена файла целиком

## Альтернативные подходы (не рекомендованы)

### Альтернатива 1: Централизованный сервер аутентификации

**Идея**: Все записи проходят через сервер, который проверяет подписи и авторизацию.

**Недостатки**:
- Нарушает принцип P2P-синхронизации через WebDAV/Dropbox
- Требует инфраструктуры (сервер, БД, мониторинг)
- Единая точка отказа

**Вердикт**: Не подходит для текущей архитектуры.

### Альтернатива 2: Блокчейн-подобный консенсус

**Идея**: Коллективное подписание снапшотов (Вариант 3 из исходного обсуждения).

**Недостатки**:
- Избыточная сложность для текстовых документов
- Требует протокола консенсуса
- Высокая нагрузка на устройства

**Вердикт**: Overkill для текущей стадии проекта.

### Альтернатива 3: Симметричное шифрование с общим ключом

**Идея**: Все устройства используют общий секретный ключ для подписи записей.

**Недостатки**:
- Не решает проблему атрибуции (кто сделал изменение)
- Любой с ключом может подписать от имени любого устройства
- Сложность распределения ключа

**Вердикт**: Не решает поставленные задачи.

## Производительность и оптимизация

### Оценка накладных расходов

| Операция | Без подписи | С подписью (Фаза 1) | Прирост |
|---|---|---|---|
| Запись записи | ~1 ms | ~5 ms (Ed25519 sign) | +4 ms |
| Чтение 100 записей | ~10 ms | ~50 ms (100 verify) | +40 ms |
| Размер записи | ~500 bytes | ~700 bytes (sig + pubKey) | +40% |

### Оптимизации

1. **Инкрементальная верификация**:
   - Кешировать проверенные записи (по `hlcKey`)
   - Верифицировать только новые записи при sync

2. **Батчинг подписей**:
   - Подписывать батч записей одной подписью (Merkle tree root)
   - Уменьшает размер, но усложняет верификацию

3. **Ленивая верификация**:
   - Верифицировать только при построении DAG
   - Не верифицировать при простом чтении для отображения

**Рекомендация**: Начать с инкрементальной верификации (кеширование), остальные оптимизации — по необходимости.

## Миграция существующих данных

### Обратная совместимость

Все изменения должны поддерживать обратную совместимость:

1. **Фаза 1**: Если запись не содержит `signature` — она считается валидной (режим «без подписи»)
2. **Фаза 2**: Если манифест отсутствует — документ считается публичным
3. **Фаза 3**: Если `seen` отсутствует — запись считается валидной
4. **Фаза 4**: Если TOFU-кеш пуст — первое наблюдение сохраняется без проверки

### Стратегия миграции

1. **Мягкий переход**: Новые записи всегда подписаны, старые записи принимаются без подписи
2. **Постепенное ужесточение**: Через N месяцев можно отклонять неподписанные записи (опционально)
3. **Уведомление пользователя**: Если обнаружены неподписанные записи — показать предупреждение

## Тестирование

### Unit-тесты

1. **Криптография**:
   - Генерация ключевой пары
   - Подпись и верификация
   - Отклонение невалидных подписей

2. **OpLogEntry**:
   - Сериализация/десериализация с новыми полями
   - Обратная совместимость (записи без подписи)

3. **OpLogDag**:
   - Верификация подписей при построении
   - Отклонение записей с невалидной подписью
   - Witness-целостность (Фаза 3)

### Integration-тесты

1. **Sync с подписями**:
   - Два устройства sync-ят документ
   - Проверка, что подписи верифицируются
   - Проверка, что подделанные записи отклоняются

2. **Атака подделки**:
   - Злоумышленник пытается дописать записи в чужой файл
   - Проверка, что подделка обнаруживается

3. **Атака усечения** (Фаза 3):
   - Злоумышленник удаляет последние записи из файла
   - Проверка, что dangling reference обнаруживается

### Security-тесты

1. **Fuzzing**:
   - Случайные изменения в `.oplog.jsonl` файлах
   - Проверка, что система не падает и отклоняет невалидные данные

2. **Replay attack** (Фаза 4):
   - Копирование подписи из одного документа в другой
   - Проверка, что подпись отклоняется (если documentPath включён в подпись)

## Заключение

Предложенная стратегия обеспечивает постепенное внедрение криптографической защиты без нарушения ключевого преимущества архитектуры — бесконфликтной файловой синхронизации.

**Фаза 1** закрывает основную угрозу (подделка записей) и может быть внедрена независимо.

**Фазы 2-4** добавляют дополнительные уровни защиты и могут внедряться по мере необходимости.

Все изменения обратно совместимы и не требуют миграции существующих данных.
